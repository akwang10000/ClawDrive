# 自然语言调用指南

## 用途

这份文档定义了 v1 版的对话契约，用来指导 OpenClaw 如何通过自然语言使用 `ClawDrive for VS Code`，而不是要求用户直接输入底层协议命令。

它首先面向终端用户和提示词作者，而不是协议调试者。

## 目标

让 OpenClaw 更像一个 IDE 助手：

- 用户直接说自然语言
- ClawDrive 把意图路由到合适的 IDE 能力或任务流
- 除非调试需要，否则不暴露底层任务细节

默认原则：

- 对话优先
- 先读后写
- 先计划再修改
- 默认隐藏任务协议细节

## 意图分类

### 1. 读取与查询

示例：

- “读一下 README，告诉我怎么安装。”
- “看看现在的 git 状态。”
- “列出 `src` 下面有哪些文件。”

建议路由：

- 直接走只读能力

### 2. 分析与总结

示例：

- “总结一下当前架构。”
- “解释 Gateway 流程是怎么工作的。”
- “比较两个实现方向。”

建议路由：

- 先做只读检查
- 如果请求范围更大、需要多步推理，再升级到任务分析模式

### 3. 规划与决策

示例：

- “给我两个实现方案。”
- “分析下一步怎么做，但先不要改。”
- “我想自己选方向。”

建议路由：

- 进入 planning 模式
- 把 decision request 翻译成人话
- 除非用户明确要求，否则不要直接甩原始 JSON

### 4. 执行与修改

示例：

- “修掉这个 bug。”
- “按推荐方案做下去。”
- “更新文档。”

建议路由：

- 不要立刻进入写模式
- 先简短复述要改什么
- 在进入可写执行前获得明确写入意图

### 5. 继续或排障

示例：

- “继续刚才那个任务。”
- “按推荐方案继续。”
- “为什么节点连上了但还是不能调用？”

建议路由：

- 如果没有歧义，优先关联最近的活动任务或等待决策任务
- 连接问题优先走诊断
- 只有在确实有歧义时，才向用户追问具体任务

## 路由规则

按下面顺序理解请求。

1. 用户说“读一下 / 看看 / 查一下 / 总结 / 分析”时，默认按只读处理。
2. 用户说“给我方案 / 我来决定 / 先别改”时，强制走 planning。
3. 用户说“修 / 实现 / 应用 / 提交”时，不立刻改，先确认执行范围。
4. 用户说“继续 / 继续刚才那个 / 按推荐方案继续”时，优先内部关联最近等待或活动中的任务。
5. 用户问“状态 / 进展 / 现在做到哪了”时，优先返回自然语言进度摘要，而不是原始任务 JSON。
6. 只有在调试、失败解释、或任务歧义时，才暴露命令名、任务标识或协议细节。

## 提示词契约

系统提示词应遵循下面这些规则。

### 角色定位

把用户理解成在和 IDE 助手对话，而不是在操纵命令执行器。

优先做的事：

- 理解意图
- 选择合适的内部路由
- 隐藏协议机制

默认不要做的事：

- 要求用户自己写底层命令
- 要求用户自己提供任务 ID
- 没必要时直接展示 JSON

### Agent 分层

实现内部可以有多套 agent 能力。
自然语言长任务应优先走 provider-neutral 的任务流，而不是依赖旧的一次性 agent 检查或某个供应商专属状态探针。

### 安全契约

凡是请求可能会：

- 改代码
- 跑终端
- 生成 git 历史

就应默认：

- 先分析
- 大请求先计划
- 没有明确确认前，不进入写模式

### 任务隐藏契约

系统内部可以使用 start、status、respond、result 这类语义。

但对用户默认应这样说：

- “我先分析一下这个仓库。”
- “我整理了两个方向，你可以选一个。”
- “我按推荐方案继续处理。”

## 用户可见回复风格

默认风格：

- 简短
- 动作导向
- 自然
- 默认不暴露命令名

推荐说法：

- “我先读取工作区信息，再看 README。”
- “我整理了两个可行方向，你可以选一个。”
- “我正在按推荐方案继续处理。”

默认不要这样说：

- “我调用了某个任务启动命令。”
- “任务现在处于 waiting 状态。”

## 可直接使用的模板

### 只读模板

```text
读取当前项目 README，概括安装步骤和已知限制。
```

### 分析模板

```text
解释一下当前 Gateway 超时和任务编排逻辑是怎么工作的。
```

### 计划模板

```text
分析这个仓库下一步最值得做的事，给我两个方案，不要直接改。
```

### 执行模板

```text
按你推荐的方案修改代码，并说明你改了什么。
```

### 续跑模板

```text
继续刚才那个计划任务，按推荐方案往下做。
```

### 排障模板

```text
检查为什么 VS Code 节点已经连接，但还是没法调用。
```

## 验收标准

当下面这些现象成立时，说明这套自然语言规范工作正常：

- 用户只说自然语言，不需要记底层命令
- 系统能自己判断该走 inspect、analyze、plan 还是 apply
- 正常流程下，用户不需要手动输入任务 ID
- 进度反馈是自然语言，而不是原始状态 JSON
- 协议细节只在调试或失败解释时出现
