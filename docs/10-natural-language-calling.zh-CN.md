# 自然语言调用指南

## 用途

这份文档定义当前 v1 阶段的对话契约，说明 OpenClaw 应如何通过自然语言使用 `ClawDrive for VS Code`，而不是要求用户自己拼装底层命令。

它优先面向终端用户和提示词作者，而不是协议调试者。

## 目标

让 OpenClaw 更像一个 IDE 助手：

- 用户直接说自然语言
- ClawDrive 把请求路由到合适的 IDE 能力
- 除非调试需要，否则默认隐藏任务协议细节

默认原则：

- 对话优先
- 先读后写
- 先规划再改动
- 默认不暴露内部任务协议

## 公共入口

当前推荐给 OpenClaw 使用的统一入口是：

- `vscode.agent.route`

这个入口接收：

- `prompt: string`
- `paths?: string[]`

然后由 ClawDrive 在内部决定是：

- 直接走只读命令
- 启动 `analyze`
- 启动 `plan`
- 继续最近相关任务

## 意图分类

### 1. 读取与查询

示例：

- “读取 README，告诉我怎么安装。”
- “列出 `src` 目录。”
- “看当前诊断。”

建议路由：

- 直接走只读命令

### 2. 分析与解释

示例：

- “解释一下这个仓库。”
- “说明 Gateway 流程是怎么工作的。”
- “比较这两个模块。”

建议路由：

- 先做只读检查
- 如果请求范围更大或需要多步推理，再升级到 `analyze`

### 3. 规划与决策

示例：

- “给我两个方案。”
- “分析下一步该怎么做，但先别改。”
- “我来决定方向。”

建议路由：

- 进入 `plan`
- 用自然语言返回方案和推荐项
- 除非用户明确要求，否则不暴露原始 JSON

### 4. 继续与排障

示例：

- “继续刚才那个任务。”
- “用推荐方案继续。”
- “为什么节点连上了但还是不可调用？”

建议路由：

- 如无歧义，优先关联最近的等待任务或活动任务
- 连接与 provider 问题优先走诊断
- 只有确实有歧义时，才要求用户选具体任务

## 当前写入规则

当前里程碑还没有开放写入型执行。

如果用户说：

- 修复
- 实现
- 修改
- 应用

当前预期行为是：

- 不直接执行写入
- 引导回 planning-first

这是一条刻意保留的 v1 安全边界，不是故障。

## 路由规则

按下面顺序理解请求：

1. 用户说“读一下 / 看看 / 列出 / 查一下 / 当前诊断”时，优先走直接只读能力。
2. 用户说“解释 / 总结 / 比较 / 怎么工作的”时，默认走 `analyze`。
3. 用户说“给我方案 / 我来决定 / 先别改”时，强制走 `plan`。
4. 用户说“继续 / 用推荐方案 / keep going”时，优先命中最近相关任务，而不是要求用户自己传 `taskId`。
5. 用户说“修复 / 实现 / apply / commit”时，不直接进入写入执行，而是挡回规划。
6. 只有在调试、失败分析或任务歧义时，才暴露命令名、任务标识或协议细节。

## 用户可见回复风格

默认风格：

- 简短
- 动作导向
- 自然
- 默认不暴露命令名

推荐说法：

- “我先检查一下当前工作区和相关文件。”
- “我整理了两个方案，你可以选一个。”
- “我已经按推荐方案继续了刚才的任务。”

默认不要这样说：

- “我调用了任务启动命令。”
- “任务现在处于 waiting 状态。”

## 可直接使用的模板

### 读取模板

```text
读取当前项目 README，并概括安装步骤和已知限制。
```

### 分析模板

```text
解释一下当前 Gateway 超时和任务编排逻辑是怎么工作的。
```

### 规划模板

```text
分析这个仓库下一步最值得做的事，给我两个方案，先别改。
```

### 继续模板

```text
继续上一个规划任务，并使用推荐方案。
```

### 诊断模板

```text
检查为什么 VS Code 节点已经连接，但还是不能调用。
```

## 验收标准

这份指南算成立，应该满足：

- 用户可以直接描述目标
- 系统能自动选择 inspect、analyze、plan 或 continue
- 正常流程不需要用户知道原始命令名
- 正常流程不需要用户知道 `taskId`
- 进度和结果默认以自然语言表达
